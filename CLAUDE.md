# Bot Factory — SaaS фабрика Telegram-ботов

## Стек

- Python 3.11, aiogram 3.x, OpenAI-compatible SDK через OpenRouter
- PostgreSQL + Redis (Docker локально, Beget в проде)
- Деплой: Docker + Beget (серверы в РФ, 152-ФЗ)
- Платежи: ЮKassa

## Структура

- main.py — Telegram intake бот
- pipeline.py — оркестратор цепочки агентов
- agents/ — аналитик, архитектор, промпт-инженер, строитель
- db/ — модели SQLAlchemy, миграции Alembic
- deployer.py — деплой в Docker
- billing.py — ЮKassa подписки

## Правила кода

- Комментарии и переменные на английском
- Секреты только через .env
- Стиль: black + isort

## Правила работы с кодом (Karpathy principles)

**Думай перед тем как кодить**
- Формулируй допущения явно, не угадывай
- При неоднозначности предлагай несколько интерпретаций
- Предлагай более простой подход если он очевиден
- Уточняй задачу если она неясна

**Простота прежде всего**
- Пиши минимум кода, решающий задачу
- Не добавляй функции которые не просили
- Не добавляй обработку ошибок для невозможных сценариев
- Если существующий код можно значительно сократить — предложи переписать

**Хирургические правки**
- Меняй только то, что требует задача
- Соответствуй существующему стилю кода, не улучшай его попутно
- Удаляй только те импорты/переменные, которые стали неиспользуемыми из-за твоих изменений
- Упоминай мёртвый код без его удаления

**Выполнение через цель**
- Переводи задачи в проверяемые критерии успеха
- «Добавь валидацию» → «напиши тесты на невалидные входы, затем сделай их зелёными»
- Перед реализацией кратко обозначь шаги и критерии проверки

## 152-ФЗ — технические требования

- Целевая инфраструктура: Beget / РФ
- PostgreSQL, Redis и файлы должны храниться на серверах в РФ
- В LLM/OpenRouter нельзя отправлять сырые персональные данные без отдельной правовой оценки
- В LLM передаётся только обезличенная техническая/анкетная модель бота; whitelist разрешённых полей — `pipeline._ALLOWED_INPUT_KEYS`
- Требуются: согласие на обработку ПДн и политика обработки персональных данных
- Примечание: это технические требования проекта, не юридическое заключение

### Data-flow карта (аудит 2026-05-05)

**Что собирается и где хранится (PostgreSQL, РФ):**
| Данные | Таблица | Примечание |
|---|---|---|
| Telegram ID | `clients.telegram_id` | Pseudonymous; обязателен для идентификации |
| Telegram username | `clients.username` | Nullable; обнуляется при /delete_my_data |
| Согласие ПДн | `clients.consent_*`, `consent_logs` | Текст согласия, дата, флаг |
| Ответы анкеты (raw) | `bot_configs.config_json.questionnaire_answers` | Хранятся полностью, не идут в LLM |
| Системный промпт | `bot_configs.system_prompt` | Сгенерирован LLM из обезличенных данных |
| История чата | `chat_history` | Контент сообщений пользователя с его же ботом |
| База знаний | `knowledge_chunks` | Загружается владельцем бота |
| Платёж (internal id) | `subscriptions.yukassa_payment_id` | Только внутренний ID платежа, не данные карты |
| Логи токенов | `token_logs` | Количество токенов + модель, без содержания |

**FSM (Redis):** временные ответы анкеты до завершения intake-flow.

**Что передаётся в OpenRouter:**
- `pipeline._ALLOWED_INPUT_KEYS` — только технические поля бота (bot_type, purpose, audience и т.д.)
- `_redact_sensitive` удаляет ответы с API-токенами и личными Telegram-контактами до отправки
- `is_sensitive_question(text, hint="")` теперь проверяет TEXT + hint; закрыты service_orders Q8, coach Q13, hr Q9 (hint "telegram username"); hr Q8 hint ("через Telegram / email") — организационный контекст, не PII контакт, оставлен без изменений
- Чат-история и RAG-чанки уходят в LLM при /chat — если пользователь написал в чате ПДн третьих лиц, они могут попасть в OpenRouter
- Embeddings (RAG): текст base знаний уходит в OpenRouter/Embeddings API

**Что передаётся в YooKassa:**
- Internal `client.id`, сумма, тариф, цикл — НЕ Telegram ID, НЕ ФИО
- YooKassa — российский оператор, отдельная правовая оценка не требуется

**Трансграничная передача:**
- OpenRouter (США): анкетные данные бота (обезличены), chat-history (может содержать ПДн третьих лиц)
- FusionBrain: текстовые промпты для генерации картинок
- Telegram API (серверы Telegram): все сообщения

**Consent-flow:**
- `/start` → CONSENT_TEXT → кнопка «Согласен» → `save_consent()` сохраняет дату и текст согласия
- `/delete_my_data` → `anonymize_user()` → обнуляет username, ставит `data_deleted=True`
- `/my_data` — ✅ реализовано (показывает: Telegram ID, username, статус согласия, кол-во ботов, подписку, кол-во сообщений чата, кол-во RAG-чанков, дату регистрации)
- `/revoke_consent` — ✅ реализовано (отзывает согласие, предлагает /delete_my_data; блокировка сервиса после отзыва — TODO)
- Ссылка на политику обработки ПДн в CONSENT_TEXT — ✅ подключена через `settings.privacy_policy_url` (опциональная)
- Ссылка на оферту в CONSENT_TEXT — ✅ подключена через `settings.terms_url` (опциональная)
- Согласие с офертой **перед оплатой** — **отсутствует** (только выбор тарифа, требует отдельной задачи)
